# Wi v0.1 architecture overview

Status: canonical for the first vertical slice  
Audience: implementers, reviewers, and future maintainers

## 1. Purpose

Wi is a personal coding-agent harness presented through a web browser. The browser is the user-facing GUI; a local backend service owns the actual work and has access to projects on disk.

The stable system shape is:

```text
web browser <-> Wi backend service <-> model provider
                         |
                         +-> project files and processes
                         +-> catalog SQLite database
                         +-> one SQLite database per session
                         +-> project services and plugins
```

The first vertical slice does not call a real model provider. It uses a deterministic fake provider to prove the backend protocol, session durability, tool loop, replay, concurrency, and failure behavior before OpenAI-specific complexity is introduced.

## 2. Product scope

Wi v0.1 is designed for:

- one operating-system user
- one local Wi backend process
- multiple registered projects
- multiple simultaneous sessions
- one active run per session
- many browser tabs or windows acting as views
- local SQLite persistence
- a browser GUI connected over HTTP and WebSocket

Wi v0.1 is not designed for:

- mutually untrusted human users
- internet-facing multi-tenant hosting
- multi-node scheduling
- distributed databases
- remote project workers
- automatic provider or billing fallbacks

## 3. Core ownership model

The backend owns durable state and work:

```text
Wi backend
  -> SessionActor
      -> Run
          -> Provider step
          -> Tool call
          -> Approval or user-input wait
```

A browser tab is only a subscriber and command source:

```text
browser tab
  -> subscribes to session events
  -> sends idempotent commands
  -> renders backend-owned state
```

Consequences:

- closing a tab does not cancel a run
- refreshing does not cancel a run
- losing a WebSocket does not cancel a run
- signing out of the Wi UI does not cancel a run
- reconnecting replays committed events from SQLite
- pending approvals and questions remain durable until resolved or cancelled

## 4. Runtime topology

```text
┌────────────────────────────────────────────────────────────┐
│ Browser                                                    │
│                                                            │
│ session list | timeline | composer | approvals | status   │
└──────────────────────┬─────────────────────────────────────┘
                       │
             HTTP      │      multiplexed WebSocket
                       │
┌──────────────────────▼─────────────────────────────────────┐
│ Wi main Node.js process                                    │
│                                                            │
│ HTTP server                                                │
│ WebSocket gateway                                          │
│ authentication/session middleware                          │
│ SessionActor registry                                      │
│ run scheduler and semaphores                               │
│ event hub and replay coordinator                           │
│ provider adapter orchestration                             │
│ plugin/project-service supervisor                          │
└───────────────┬────────────────┬───────────────────────────┘
                │                │
                │ worker RPC     │ child-process / worker RPC
                ▼                ▼
       ┌────────────────┐   ┌──────────────────────────────┐
       │ SQLite workers │   │ Tools and project services   │
       │                │   │                              │
       │ catalog worker │   │ fake tools in first slice    │
       │ session pool   │   │ real tools later             │
       └────────────────┘   └──────────────────────────────┘
                │
                ▼
       catalog.sqlite3 + per-session session.sqlite3 files
```

The main Node.js event loop is a control plane. Synchronous SQLite, CPU-heavy work, and arbitrary plugin code do not run there.

## 5. Browser communication

Wi uses both HTTP and WebSocket.

HTTP is used for:

- serving the browser application
- bootstrap data
- health checks
- browser authentication establishment
- future large file/blob upload and download endpoints

One multiplexed WebSocket per browser client is used for:

- commands
- command acknowledgements
- session subscriptions
- event replay
- live session events
- approvals
- cancellation
- pending-input responses
- heartbeats

Browser SSE is not part of v0.1.

## 6. Persistence model

Wi stores:

```text
$WI_HOME/
  catalog.sqlite3
  sessions/
    <prefix>/
      <session-id>/
        session.sqlite3
        artifacts/
  logs/
  tmp/
```

The catalog database is always available and maps session IDs to dedicated database paths. It stores only summary/index information needed to list and locate sessions.

Each session database is canonical for that session and contains:

- immutable ordered events
- accepted command records
- run projections
- messages and parts
- provider-step projections
- tool-execution ledger
- approvals and pending inputs
- session manifest and schema version

The event history is logically append-only. SQLite files themselves are not required to be physically append-only.

## 7. Concurrency model

Concurrency is organized by session:

```text
Session A -> one actor -> one active run -> A.sqlite3
Session B -> one actor -> one active run -> B.sqlite3
Session C -> one actor -> one active run -> C.sqlite3
```

Different sessions may run concurrently. Commands for one session are serialized by that session's actor and database-worker assignment.

Global limits bound:

- active provider operations
- active tool executions
- database-worker count
- CPU-worker count
- browser outbound queue sizes

The first vertical slice proves this using the fake provider and safe test tools.

## 8. Provider strategy

The provider boundary is transport- and vendor-neutral.

First vertical slice:

```text
provider-fake -> provider-contract
```

Later direct adapters:

```text
openai_platform
  -> OpenAI Platform API key
  -> public Responses API

openai_codex
  -> ChatGPT OAuth access/refresh tokens
  -> ChatGPT Codex Responses compatibility endpoint
```

Wi will not invoke `codex app-server`, and it will not silently switch account, endpoint, provider, model, transport, or billing source.

## 9. Tool strategy

Every tool passes through:

```text
tool registry
  -> schema validation
  -> policy and approval
  -> durable tool ledger
  -> execution
  -> durable result
  -> provider continuation
```

The first slice includes only deterministic tools such as `echo`, `guarded_echo`, and a cancellable `delay`.

Real filesystem and shell tools are deferred until the durable tool semantics are proven.

## 10. Project and filesystem strategy

The active project is the default working directory and context root. For the personal installation, future real tools will use a host-unrestricted filesystem policy rather than treating the project root as a hard sandbox boundary.

Wi's own credentials and critical live internal state remain inaccessible to generic model tools. This protects authentication material without preventing deliberate access to other user-selected directories.

No unrestricted filesystem tool is implemented in the first vertical slice.

## 11. Plugin strategy

Wi standardizes the tool contract, lifecycle, events, policy, cancellation, and version snapshots. It does not force every plugin to use the same internal runtime mechanism.

Planned examples:

- CodeGraph: project-scoped service using its public library API in a worker or isolated service
- Context Mode: a lean, project-isolated adapter exposing only selected tools

Plugins never execute uncontrolled work on the main server event loop.

Plugins are excluded from the first vertical slice.

## 12. Principal invariants

1. A durable event is published only after its session transaction commits.
2. The session database is canonical; the catalog is rebuildable.
3. Session events are immutable and monotonically ordered.
4. Browser connection lifetime does not own run lifetime.
5. One active run exists per session.
6. Every state-changing command is idempotent.
7. A tool never executes from a partial or nonterminal provider response.
8. Every tool effect has a durable ledger record.
9. Synchronous SQLite never runs on the main event loop.
10. Failures remain within the smallest reasonable fault domain.
11. No hidden provider or `codex app-server` fallback exists.

## 13. First vertical-slice end-to-end flow

```text
1. Browser creates a session.
2. Backend reserves and initializes its dedicated database.
3. Browser submits an idempotent message command.
4. SessionActor commits the user message and run creation.
5. Backend acknowledges only after commit.
6. Fake provider emits provisional text.
7. Fake provider emits a complete staged tool call.
8. Fake provider reaches a valid terminal event.
9. Backend promotes and commits the tool request.
10. Tool executes exactly once through the ledger.
11. Tool result is committed.
12. Fake provider continues and produces a final answer.
13. Run completion is committed and published.
14. Browser disconnects and reconnects.
15. Missing events replay exactly from the last sequence.
```

## 14. Package boundaries

```text
protocol
  browser-safe wire and domain schemas

storage
  SQLite schemas, workers, repositories, migrations, recovery

provider-contract
  provider adapter interface and normalized provider events

provider-fake
  deterministic provider implementation and scenarios

tools
  tool registry, policy, execution, and ledger abstractions

harness-core
  actors, scheduler, run loop, replay coordination, cancellation

client-state
  pure browser reducer and replay logic

apps/server
  composition root, HTTP, WebSocket, local auth, logging

apps/web
  browser GUI
```

## 15. Acceptance boundary

The first vertical slice is accepted only when automated tests demonstrate:

- concurrent sessions
- one active run per session
- exact command idempotency
- commit-before-publication
- browser-independent runs
- race-free replay
- tool execution only after provider completion
- tool-call deduplication
- durable approvals across restart
- process-level crash recovery at documented failpoints
- slow-consumer isolation
- malformed-protocol resilience
- property and fuzz invariants

OpenAI integration begins only after this boundary is green.
