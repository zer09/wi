# Wi v0.1.0 — First Vertical Slice Implementation Plan for Codex

Status: implementation-ready  
Target runtime: Node.js 24 LTS on Linux, TypeScript, ESM, pnpm workspaces
Scope: local, trusted single operating-system user; multiple projects and concurrent sessions
Provider in this slice: deterministic fake provider only  
Browser transport: HTTP for bootstrap/static/blob endpoints; one multiplexed WebSocket for commands and events  
Persistence: one catalog SQLite database plus one SQLite database per session  
Canonical history: append-only session events with transactional projections  
Explicitly excluded: Windows/macOS server support, hostile concurrent same-user mutation of `WI_HOME`, OpenAI Platform integration, ChatGPT/Codex OAuth, `codex app-server`, SSE to the browser, CodeGraph, Context Mode, real shell/file mutation tools, remote/multi-user deployment

---

## 1. Goal

Build the smallest end-to-end implementation that proves the hardest architectural invariants:

```text
browser
  -> WebSocket command
  -> backend command validation
  -> SessionActor
  -> per-session SQLite transaction
  -> fake provider stream
  -> completed tool call
  -> tool ledger and tool execution
  -> fake provider continuation
  -> committed session events
  -> WebSocket live delivery
  -> browser disconnect
  -> backend run continues
  -> browser reconnect
  -> exact event replay
```

The slice is complete only when two or more sessions can run concurrently, a browser tab can disappear without cancelling a run, committed events can be replayed exactly, duplicate commands cannot duplicate effects, partial provider tool calls cannot execute, and process-level crash tests demonstrate the expected recovery semantics.

## 2. Non-negotiable invariants

Codex must preserve these invariants throughout implementation.

### 2.1 Backend ownership

- Runs belong to the Wi backend, never to a browser tab or WebSocket.
- Closing, refreshing, disconnecting, or replacing a browser tab does not cancel a run.
- A run is cancelled only by an explicit durable cancellation command, server shutdown policy, or unrecoverable backend/provider failure.

### 2.2 Commit before publication

No durable event is sent to a browser until the session database transaction containing that event and its projection updates has committed.

```text
validate
  -> begin transaction
  -> append event(s)
  -> update projections
  -> commit
  -> publish
```

A crash after commit but before publication is recovered through replay. A crash before commit must leave no visible event.

### 2.3 Session database is canonical

- The dedicated session database is the source of truth.
- The catalog database is a rebuildable index/projection.
- Cross-database atomicity must never be required.
- Catalog updates happen after a session commit and may temporarily lag.
- Every session database contains enough manifest data to reconstruct its catalog entry.

### 2.4 Append-only canonical events

- Event rows cannot be updated or deleted through normal application code.
- Database triggers reject event update and delete operations.
- Each session has one monotonically increasing `sequence`.
- Event IDs are globally unique, but ordering is determined only by session sequence.

### 2.5 One writer per session

- Commands and run-state mutations for one session are serialized by its SessionActor and database worker assignment.
- Different sessions may progress concurrently.
- Only one active agent run is allowed per session in v0.1.0.

### 2.6 Main event loop remains orchestration-only

The main Node.js event loop may perform:

- HTTP and WebSocket handling
- validation
- scheduling
- lightweight state transitions
- bounded serialization
- provider network I/O

It must not perform:

- synchronous SQLite work
- CPU-heavy parsing or analysis
- synchronous child-process calls
- unbounded JSON processing
- plugin execution

All SQLite access runs inside worker threads.

### 2.7 Idempotent browser commands

- Every state-changing browser command has a `commandId`.
- A repeated command with the same ID and identical canonical payload returns the original result.
- Reusing the same ID with a different method or payload is a conflict and must not execute either payload again.
- A command is acknowledged only after its acceptance transaction commits.

### 2.8 Safe provider partial-stream behavior

- Provider text may be stored and displayed as provisional streaming output.
- Tool calls are staged while a provider response is streaming.
- No tool executes until the provider response reaches a valid accepted terminal event.
- A response that fails, is cancelled, ends without a terminal event, or is incomplete cannot initiate a new tool effect.
- Partial text remains visible and is marked interrupted.

### 2.9 Tool effect ledger

- Every tool call is recorded before execution.
- Duplicate `callId` plus identical arguments reuses the existing execution/result.
- Duplicate `callId` plus different arguments is a provider protocol error.
- The initial `echo` tool is pure and retryable.
- A started tool with no committed result is reconciled according to its effect class at startup.

### 2.10 No hidden fallback

- Do not implement or invoke `codex app-server`.
- Do not implement OpenAI or ChatGPT authentication in this slice.
- Do not silently switch provider, endpoint, transport, model, or billing source.
- Provider failures fail the smallest reasonable fault domain and emit a safe browser error plus a detailed redacted log.

### 2.11 Canonical ADR numbering

The canonical ADR index is `docs/adr/README.md`.

Earlier architecture discussion used temporary conversational labels ADR-009 through ADR-014. They are reconciled as follows rather than duplicated:

```text
conversation ADR-009 -> canonical ADR-0006 (no app-server fallback)
conversation ADR-010 -> canonical ADR-0004 (backend-owned runs)
conversation ADR-011 -> canonical ADR-0007 (partial-stream policy)
conversation ADR-012 -> canonical ADR-0008 (tool-effect ledger)
conversation ADR-013 -> canonical ADR-0009 (host-unrestricted filesystem)
conversation ADR-014 -> canonical ADR-0010 (plugin-specific integration)
```

ADR-0009 and ADR-0010 are accepted future boundaries but their real filesystem/plugin behavior remains excluded from the first vertical slice.

## 4. Recommended technical choices

Use these choices unless a concrete compatibility issue is demonstrated.

- Node.js 24 LTS
- TypeScript with `strict`, `noUncheckedIndexedAccess`, and `exactOptionalPropertyTypes`
- ESM packages
- pnpm workspaces
- Vitest for unit and integration tests
- fast-check for property-based/fuzz tests
- Playwright for browser end-to-end tests
- Zod for runtime wire-message and worker-RPC validation
- Fastify for HTTP
- `@fastify/websocket` or `ws` for WebSocket handling, with framework types contained inside `apps/server`
- React plus Vite for the minimal browser application
- `better-sqlite3` behind a narrow internal adapter, but only inside worker threads
- Pino-compatible structured JSON logging with mandatory redaction
- Node `worker_threads` for catalog and session database workers

Do not let a third-party library's types become the canonical domain model. Wrap all libraries at package boundaries.

## 6. Wire protocol v1

### 6.1 General envelope

All messages use JSON and include:

```ts
interface WireEnvelope {
  v: 1;
  kind: string;
}
```

Reject:

- unknown protocol versions
- unknown message kinds
- unknown required command methods
- non-object JSON values
- messages beyond the configured byte limit
- duplicate object keys if the selected parser can detect them
- invalid UTF-8 at the WebSocket layer

### 6.2 Client messages

```ts
type ClientMessage =
  | HelloMessage
  | SubscribeMessage
  | UnsubscribeMessage
  | CommandMessage
  | ClientHeartbeatMessage;
```

#### Hello

```ts
interface HelloMessage {
  v: 1;
  kind: "hello";
  clientId: string;
  resume: Array<{
    sessionId: string;
    afterSequence: number;
  }>;
}
```

#### Subscribe

```ts
interface SubscribeMessage {
  v: 1;
  kind: "subscribe";
  requestId: string;
  sessionId: string;
  afterSequence: number;
}
```

#### Command

```ts
interface CommandMessage {
  v: 1;
  kind: "command";
  commandId: string;
  sessionId?: string;
  method:
    | "session.create"
    | "message.submit"
    | "run.cancel"
    | "approval.resolve"
    | "input.respond";
  params: unknown;
}
```

### 6.3 Server messages

```ts
type ServerMessage =
  | WelcomeMessage
  | CommandAcceptedMessage
  | CommandRejectedMessage
  | EventMessage
  | ReplayCompleteMessage
  | ProtocolErrorMessage
  | ServerHeartbeatMessage;
```

#### Durable event message

```ts
interface EventMessage {
  v: 1;
  kind: "event";
  sessionId: string;
  sequence: number;
  eventId: string;
  eventType: SessionEventType;
  createdAtMs: number;
  data: unknown;
}
```

### 6.4 Command acknowledgement rules

`command.accepted` is sent only after database commit. It includes:

```ts
interface CommandAcceptedMessage {
  v: 1;
  kind: "command.accepted";
  commandId: string;
  sessionId?: string;
  acceptedSequence?: number;
  runId?: string;
  result: unknown;
  duplicate: boolean;
}
```

A duplicate command with identical canonical content returns the stored acceptance result with `duplicate: true`.

### 6.5 Canonical command hashing

Implement deterministic canonical JSON for command identity:

- object keys sorted lexicographically
- arrays retain order
- numbers must be finite
- distinguish `null` from missing
- reject `undefined`, functions, symbols, bigint, NaN, and infinities
- UTF-8 encode the canonical representation
- hash with SHA-256

Store:

```text
command_id
method
payload_hash
result_json
accepted_sequence
run_id
accepted_at_ms
```

```sql

    project_id TEXT,
    db_relative_path TEXT NOT NULL UNIQUE,
    title TEXT NOT NULL DEFAULT '',
    status TEXT NOT NULL,
    created_at_ms INTEGER NOT NULL,
    updated_at_ms INTEGER NOT NULL,
    last_event_sequence INTEGER NOT NULL DEFAULT 0,
    last_run_state TEXT,
    last_message_preview TEXT,
    requires_attention INTEGER NOT NULL DEFAULT 0,
    pending_approval_count INTEGER NOT NULL DEFAULT 0,
    pending_input_count INTEGER NOT NULL DEFAULT 0,
    session_schema_version INTEGER NOT NULL
) STRICT;

CREATE INDEX sessions_updated_idx
    ON sessions(updated_at_ms DESC);

CREATE TABLE catalog_commands (
    command_id TEXT PRIMARY KEY,
    command_method TEXT NOT NULL,
    payload_hash TEXT NOT NULL,
    state TEXT NOT NULL,
    reserved_session_id TEXT,
    result_json TEXT,
    accepted_at_ms INTEGER,
    updated_at_ms INTEGER NOT NULL
) STRICT;
```

`catalog_commands` handles idempotency for global commands such as `session.create`, because no dedicated session database exists when that command first arrives. Session-scoped commands use the session database's `accepted_commands` table.

Session creation uses a recoverable state machine:

```text
reserve command + session ID/path in catalog (`creating`)
  -> create and initialize session database
  -> commit `session.created` in session database
  -> mark catalog session and catalog command `ready/accepted`
  -> acknowledge browser
```

A duplicate `session.create` with the same command ID and payload returns the reserved or completed session result. Reusing the command ID with a different payload is a conflict. Startup recovery completes or cleans up catalog commands left in `creating`.

The project table may use a temporary test project during the first slice. Do not implement unrestricted filesystem tools yet.

## 9. Database worker architecture

### 9.1 Catalog worker

One dedicated worker thread owns the catalog connection for the lifetime of the server.

Main-thread API examples:

```ts
interface CatalogClient {
  createSession(input: CreateSessionInput): Promise<CreateSessionResult>;
  listSessions(): Promise<SessionSummary[]>;
  getSession(sessionId: string): Promise<SessionSummary | null>;
  updateSessionProjection(update: CatalogSessionUpdate): Promise<void>;
  reconcileSession(manifest: SessionManifest): Promise<void>;
  close(): Promise<void>;
}
```

### 9.2 Session worker pool

Use a fixed worker pool. Do not create one worker thread per session.

```text
workerIndex = stableHash(sessionId) % poolSize
```

Each worker:

- owns an LRU of open session database connections
- lazily opens a connection on first use
- serializes operations for each assigned session
- closes idle handles
- never receives arbitrary SQL from the main thread
- exposes named, validated operations only

Suggested default:

```text
development/test: 2 session DB workers
production default: clamp(cpuCount - 1, 2, 4)
```

Make it configurable.

### 9.3 Worker RPC

Every message has:

```ts
interface WorkerRequest {
  requestId: string;
  operation: string;
  payload: unknown;
}

interface WorkerResponse {
  requestId: string;
  ok: boolean;
  result?: unknown;
  error?: {
    code: string;
    message: string;
    retryable: boolean;
  };
}
```

Validate both requests and responses. A malformed worker response is a worker protocol failure; terminate and replace that worker, and fail only affected operations.

### 9.4 SQLite configuration

On each connection:

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = FULL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
PRAGMA trusted_schema = OFF;
```

Also:

- use prepared statements
- use strict tables
- disable extension loading
- apply migrations before normal operations
- inspect `sqlite_version()` at startup, log it, and fail startup if it is below the project's configured minimum (initially SQLite 3.51.3)
- never concatenate untrusted values into SQL
- set practical payload limits in application code

## 10. SessionActor design

### 10.1 Actor responsibilities

A SessionActor owns:

- command serialization for one session
- one active run
- queued follow-up messages
- pending approval/input state
- run cancellation controller
- interaction with the session storage client
- interaction with the run scheduler
- event publication after commits
- replay subscription coordination

It does not own:

- browser connection lifetime
- SQLite connection objects
- provider credentials
- global scheduling limits
- project service implementation

### 10.2 Mailbox

The mailbox must process short state transitions only. Never await an entire provider run inside the mailbox.

Bad:

```ts
await executeCompleteRun();
```

## 11. Provider contract and fake provider

## 11.1 Provider contract

```ts
interface ProviderAdapter {
  readonly id: string;

  stream(
    request: ProviderRequest,
    context: ProviderContext,
    signal: AbortSignal,
  ): AsyncIterable<ProviderEvent>;
}
```

Normalized provider events:

```ts
type ProviderEvent =
  | { type: "response.started"; responseId: string }
  | { type: "text.delta"; delta: string }
  | {
      type: "tool_call.completed";
      callId: string;
      name: string;
      argumentsJson: string;
    }
  | { type: "response.completed"; responseId: string }
  | {
      type: "response.failed";
      category: string;
      message: string;
      retryable: boolean;
    };
```

A stream ending without `response.completed` or `response.failed` is a protocol interruption.

## 11.2 Scripted fake provider

The fake provider accepts a deterministic script:

```ts
interface FakeProviderScript {
  steps: FakeProviderStep[];
}
```

Required scenarios:

1. `plain-text`
2. `echo-tool-round-trip`
3. `approval-round-trip`
4. `slow-stream`
5. `cancel-before-output`
6. `transient-failure-before-output`
7. `failure-after-visible-output`
8. `partial-tool-call-without-terminal`
9. `duplicate-call-id-same-arguments`
10. `duplicate-call-id-different-arguments`
11. `stream-closes-without-terminal`
12. `provider-never-completes-until-aborted`

Tests must select scripts explicitly. Production/dev behavior must not infer privileged scenarios from arbitrary user text.

The chosen fake-provider scenario/configuration is persisted in `runs.provider_config_json`. Together with the persisted message/tool history and provider step index, this must be sufficient to reconstruct a run that was waiting for approval when the server restarts.

## 11.3 Tool-call staging

Completed tool-call items must be durable but non-executable until the provider step terminates successfully.

During one provider step:

1. Persist/coalesce text deltas.
2. When a complete tool-call item arrives, validate its basic envelope and insert/update its ledger projection with state `staged`; append `provider.tool_call.staged`.
3. Do not execute it yet.
4. Wait for the provider terminal event.
5. On `response.completed`, use one session transaction to:
   - mark the provider step completed
   - validate all staged calls against the registered tool schemas
   - promote valid staged calls to `requested`
   - append `provider.step.completed` and `tool.call.requested` events
6. Publish those committed events.
7. Execute approved calls through the ledger.
8. Add committed tool results to the next provider request.
9. Start the next provider step.
10. Complete when a successful provider step ends without tool calls.

On provider failure, cancellation, stream closure without a terminal event, or incomplete response, use one transaction to mark the step interrupted/failed and mark staged calls `discarded`. Append interruption events and never execute those calls.

This makes crash recovery deterministic:

- crash before terminal event: startup marks streaming step interrupted and staged calls discarded
- crash during the successful terminal transaction: either all promotions commit or none do
- crash after promotion but before execution: recovery sees `requested` calls and can safely continue

## 12. Tool registry and execution ledger

### 12.1 Tool definition

```ts
interface ToolDefinition<TInput, TOutput> {
  name: string;
  description: string;
  inputSchema: ZodType<TInput>;
  effectClass: "pure" | "local_transactional" | "idempotent_external" | "non_idempotent";
  approval: "never" | "always";
  timeoutMs: number;

  execute(
    input: TInput,
    context: ToolExecutionContext,
    signal: AbortSignal,
  ): Promise<TOutput>;
}
```

### 12.2 Initial tools

#### `echo`

- pure
- no approval
- returns input text and deterministic metadata
- increments a test-visible execution counter through injected test instrumentation, not global production state

#### `guarded_echo`

- pure
- always requires approval
- otherwise identical to echo
- exists to prove durable waiting/approval behavior

#### `delay`

- pure
- no approval
- cancellable
- used to prove browser disconnect independence, explicit cancellation, and scheduler limits

### 12.3 Duplicate execution behavior

Inside a transaction:

- absent call ID: insert `requested`
- same call ID and same tool/argument hash:
  - completed: return stored result
  - failed/denied: return stored terminal result
  - requested/approved: continue existing state
  - started: apply recovery policy
- same call ID with different tool or argument hash: provider protocol error

### 12.4 Crash recovery for pure tools

If startup finds `effect_class = pure` and `state = started` with no result:

- append a recovery event
- reset to an executable retry state
- increment attempt count on retry
- never create a second ledger row

This is acceptable only because initial tools are pure.

## 13. WebSocket gateway

### 13.1 Connection model

- one WebSocket per browser tab/client
- one socket can subscribe to multiple sessions
- one session can have multiple subscribers
- a socket does not own a run
- disconnection removes only the subscriber

### 13.2 Replay algorithm

To avoid the replay/live race:

1. Register subscriber as `replaying`.
2. Ask SessionActor/storage for current head sequence `H`.
3. Query committed events `(afterSequence, H]`.
4. Queue live events `> H` for that subscriber while replay is sent.
5. Send replay events in order.
6. Send `replay.complete` with `throughSequence: H`.
7. Drain queued live events in order.
8. Mark subscriber `live`.

The client reducer must tolerate an exact duplicate sequence but must detect a gap.

### 13.3 Outbound backpressure

Each connection has bounded:

- queued message count
- queued bytes
- maximum single message size

Policy:

1. coalesce low-priority text deltas
2. never drop terminal state, approval, error, or tool-state events
3. if still above limit, close the slow connection with an application close code
4. the client reconnects and replays from its last applied sequence

### 13.4 Heartbeats

- server protocol ping/pong detects dead sockets
- optional application heartbeat communicates server time
- heartbeat events are not stored in session databases

### 13.5 Local security

For this slice:

- bind to `127.0.0.1` by default
- reject unexpected `Host`
- validate WebSocket `Origin`
- issue an HttpOnly same-site local browser-session cookie from bootstrap
- require that cookie for WebSocket upgrade
- use a strict Content Security Policy
- render timeline text as text, never raw HTML
- cap inbound message bytes before JSON parsing
- redact cookie and authorization values from logs

Security must not add approval prompts to normal local use.

## 14. Browser application

The browser UI is intentionally minimal.

### 14.1 Required views

- connection state
- session list
- create session
- select/open session
- timeline
- message composer
- current run state
- cancel action
- pending approval panel

### 14.2 Client state rules

Use one pure reducer for both replayed and live events.

```ts
nextState = reduceSessionEvent(previousState, event);
```

Per session, track:

```ts
interface BrowserSessionState {
  lastAppliedSequence: number;
  status: "loading" | "replaying" | "live" | "gap" | "error";
  timeline: TimelineItem[];
  activeRun?: BrowserRunState;
  pendingApprovals: Record<string, BrowserApproval>;
}
```

Sequence handling:

- `sequence === last + 1`: apply
- `sequence <= last`: ignore exact duplicate
- `sequence > last + 1`: mark gap and resubscribe from `last`
- do not optimistically invent durable run state before command acceptance

### 14.3 Reconnect

Use bounded exponential backoff with jitter. On reconnect:

- send `hello`
- include last applied sequence for every open session
- do not resubmit accepted commands
- only resubmit commands that never received an acceptance/rejection, using the same `commandId`

# 20. Codex working protocol

Codex should follow this process for every milestone.

1. Read root `AGENTS.md`, this plan, and relevant ADRs.
2. Inspect existing code and tests before editing.
3. Restate the milestone scope and list files expected to change.
4. Implement only that milestone.
5. Add tests before or alongside production behavior.
6. Run the narrow test set repeatedly while developing.
7. Run `pnpm check` before declaring the milestone complete.
8. Report:
   - files changed
   - invariants implemented
   - tests added
   - exact commands run
   - remaining risks
9. Stop at the milestone boundary.
10. Do not silently reinterpret architecture. Record a proposed ADR change when necessary.

A milestone is not complete when code merely compiles. Its exit gate must pass.

# 21. Suggested root AGENTS.md

```markdown
# Wi repository instructions

## Scope

Wi is a local, single-user, browser-based coding-agent harness.

The browser is a temporary GUI. The Node backend owns sessions, runs,
provider requests, tools, approvals, and persistence.

## Runtime and tooling

- Use Node.js 24 LTS.
- Use pnpm workspaces.
- Use TypeScript ESM with strict checking.
- Run `pnpm check` before completing a milestone.
- Add unit, integration, property, and process tests as required by the plan.

## Hard architectural invariants

- Commit session events before publishing them.
- The per-session database is canonical; the catalog is rebuildable.
- Canonical session events are append-only.
- One active run per session; many sessions may run concurrently.
- Browser disconnect never cancels a run.
- Every state-changing browser command is idempotent by command ID.
- Do not execute a tool from a partial or nonterminal provider response.
- Every tool effect passes through the durable tool ledger.
- Do not run synchronous SQLite on the main event loop.
- Do not import web-framework types into protocol or harness-core packages.
- Do not invoke `codex app-server`.
- Do not add OpenAI integration during the first vertical slice.
- Do not add browser SSE; use the multiplexed WebSocket protocol.
- Fail the smallest reasonable fault domain and log a redacted diagnostic.

## Dependency boundaries

- `protocol` is dependency-light and browser-safe.
- `storage` owns SQLite implementation details.
- `harness-core` depends on interfaces, not Fastify, React, or SQLite drivers.
- `provider-fake` implements `provider-contract`.
- `client-state` contains a pure reducer usable for replay and live events.
- `apps/server` composes backend packages.
- `apps/web` is a GUI and never owns backend run state.

## Testing

- Persist fast-check seeds and minimized counterexamples.
- Use deterministic clocks and ID generators in tests.
- Include real process-restart tests for crash windows.
- A test must fail if an event is published before commit.
- A test must fail if a duplicate command creates a duplicate run or tool effect.
```

# 23. Definition of success

The first vertical slice succeeds when Wi has demonstrated, with automated tests rather than assumptions, that:

- browser tabs are disposable views
- backend runs survive browser disconnects
- multiple sessions progress concurrently
- each session has an independent lazily opened database
- the catalog lists sessions without opening all session databases
- events are append-only and replayable
- commands are idempotent
- provider partial streams cannot trigger tools
- tool effects are ledgered and deduplicated
- approvals are durable and single-resolution
- crashes have deterministic recovery semantics
- slow clients do not block the backend
- malformed input does not crash the server
- the browser state reconstructed from replay equals the live state
- no hidden provider or app-server fallback exists

Only after this gate should the project add the direct OpenAI Platform adapter and then the direct ChatGPT/Codex OAuth adapter.
